# OpenClaw 沙箱：工具执行流程 (exec)

OpenClaw 通过将宿主机的指令透明地转发并包装，实现在沙箱环境内安全运行任意代码。

## 1. 通信机制：指令下达与流式反馈

Gateway 与沙箱之间通过“指令即时下达、结果实时回传”的模式进行通信。这种交互主要在三个维度展开：

### 维度一：指令触发与管道通信 (主通道 - `docker exec`)
这是系统最基础的控制平面。当 Gateway 接收到 Agent 的工具请求时，会利用 Node.js 的 `spawn` 机制在宿主机上启动一个新的 Docker 进程：
*   **指令包装**：执行底层的 `docker exec -i [容器名] /bin/sh -lc "[原始指令]"`。
*   **三条水管 (Pipes)**：在 `src/agents/sandbox/docker.ts` 中，系统通过 `stdio: ["pipe", "pipe", "pipe"]` 在宿主机与沙箱间架设了双向数据链路：
    *   **输入流 (stdin)**：Gateway 将复杂的代码块、配置文件或脚本内容直接“喂”进沙箱进程。
    *   **输出流 (stdout/stderr)**：沙箱内程序运行的所有日志、计算结果和报错信息，都会实时“流”回宿主机。Gateway 监听并解析这些数据流。

### 维度二：文件共享 (数据通道 - Bind Mounts)
为了避免海量数据的网络传输，Gateway 与沙箱共享物理存储区域：
*   **映射逻辑**：宿主机的项目工作区被挂载至容器的 `/workspace`。
*   **交互方式**：Gateway 将大文件写入宿主机目录，沙箱内的工具进程立即在映射路径下可见并进行处理；处理产物写回，Gateway 实时感知。

### 维度三：网络桥接 (专用通道 - Browser Bridge)
针对 `browser` 工具等高交互场景，系统采用更高级的桥接技术：
*   **CDP 映射**：沙箱内的浏览器开启 CDP 端口（默认 9222）。
*   **端口转发**：Docker 引擎将容器端口动态映射至宿主机回环地址。
*   **WS 通信**：Gateway 内置的 Bridge Server 通过 WebSocket 与沙箱浏览器建立高速双向链路。

## 2. 完整的通信链路示例
假设用户提问：“帮我算一下 1+1 等于几？”

1.  **Gateway (宿主机)**：接收问题并投递给远端 LLM。
2.  **LLM (决策)**：回复 Gateway：“我要运行 Python 脚本 `print(1+1)`”。
3.  **Gateway (宿主机)**：
    *   构造指令：`docker exec -i openclaw-sbx-agent1 python3`。
    *   通过 **stdin 管道** 将代码字符串发送过去。
4.  **沙箱 (容器内)**：Python 解释器运行代码，在内部输出 `2`。
5.  **Gateway (宿主机)**：通过 **stdout 管道** 接收到字符串 `"2"` 并告知 LLM。
6.  **LLM & 用户**：LLM 最终基于结果回复用户。

## 3. 核心技术细节

### 登录 Shell (`-lc`)
强制使用 `-lc` 参数至关重要：
*   **`-l` (login)**：确保沙箱容器内的 `/etc/profile`、`~/.bashrc` 或用户特定的环境变量被完整加载。
*   **`-c` (command)**：执行随后的指令字符串。
这一设计保证了沙箱内的开发环境（如 `nvm` 配置的 Node 版本）与宿主机感知的一致。

### 环境变量注入
*   **显式传递**：通过 `docker exec -e KEY=VALUE` 将宿主机的关键非敏感上下文（如时区、语言设置）注入。
*   **PATH 优先级**：系统会特殊处理 `PATH` 变量，确保沙箱内的专用工具链（如位于 `/usr/local/bin/` 下的隔离工具）优先于通用系统命令。

## 3. 标准流交互
*   **`stdin` 绑定**：支持交互式命令输入（通过 `-i` 参数）。
*   **输出捕获**：网关实时捕获 `stdout` 和 `stderr`，并在返回给 AI 前进行必要的格式清洗或隐私过滤。

## 4. 路径对齐
系统内置路径转换逻辑。虽然 AI 在沙箱内执行命令，但它感知的路径通常是容器映射后的 `/workspace`。网关负责在指令发送前将路径翻译为容器视角，在结果返回后翻译回宿主机视角。
