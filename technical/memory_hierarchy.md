# OpenClaw 记忆归属与层级架构

在 OpenClaw 的多代理架构中，长期记忆的归属权主要取决于其 **“工作区 (Workspace)”** 的配置。系统允许用户构建一个知识部分共享，但性格与操作指南完全独立的“代理矩阵”。

## 1. 物理隔离：工作区目录 (Workspace Directory)

在 `config.json` 中，可以为每个 `agentId` 设置独立的 `workspace` 路径：
*   **完全隔离模式**：若 Agent A 与 Agent B 指向不同的物理路径（如 `~/memories/agent-a` 与 `.../agent-b`）。
    *   它们拥有各自独立的 `AGENTS.md`、`SOUL.md` 和 `memory/` 日志文件夹。
    *   **结果**：Agent A 积累的实战教训，Agent B 完全无法感知。它们是互不相识的独立个体。
*   **根目录共享模式**：这是默认情况（通常为 `~/.openclaw/workspace`）。
    *   多个 Agent 共享同一根目录，通过文件名后缀或子文件夹前缀区分特定记忆。
    *   它们会共同读取同一份 `USER.md`，确保对主人的认知保持一致。

## 2. 逻辑隔离：文件级定义

即使在同一个物理工作区内，通过加载不同的核心文件也能实现差异化：
*   **人格独立性**：系统支持为不同代理加载特定灵魂文件（如 `SOUL.coder.md` 与 `SOUL.life.md`），在启动时根据 `agentId` 动态注入。
*   **经验演化差异**：随着时间推移，专门负责代码的 Agent 会在 `AGENTS.md` 中积累大量技术避坑指南，而负责日程的 Agent 则会积累大量的交互习惯，形成各具特色的“大脑”。

## 3. 子代理 (Sub-agent) 的记忆流转

当主代理通过 `sessions_spawn` 启动子代理时：
*   **瞬时隔离**：子代理通常被赋予临时工作区或主工作区的受限子集。它能看到当前任务指令，但默认无法访问主代理的所有历史私密日记。
*   **结果回传 (Memory Upstreaming)**：子代理任务结束后，其重要发现会被“通报 (Announce)”给主代理。主代理随后决定是否将这些发现持久化到自己的长期记忆中。
*   **本质**：子代理的短期记忆通过主代理的筛选，转化为主代理的长期经验。

## 4. 共享的“主人记忆” (USER.md)

无论 Agent 如何变化，关于 **“用户是谁”** 的记忆通常是全局共享的：
*   **全局性**：所有 Agent 默认读取同一个 `USER.md`。
*   **目的**：确保无论用户与哪个 Agent 交互，AI 都清楚用户的姓名、地理位置、语言偏好及特定禁忌。

## 5. 记忆权属总结对照表

| 记忆组件 | 独立性 | 典型内容 |
| :--- | :--- | :--- |
| **`AGENTS.md`** | **高度独立** | 专业领域的操作手册、实战避坑教训。 |
| **`SOUL.md`** | **高度独立** | 独特的性格人设、语气风格、道德边界。 |
| **`USER.md`** | **通常共享** | 用户背景、生活偏好、安全授权习惯。 |
| **`memory/*.md`** | **可配置** | 每日情节日志，可按 Agent 编号隔离。 |
| **向量数据库** | **按 Agent 索引** | 语义搜索时，系统通常根据当前 `agentId` 进行过滤。 |

## 6. 核心哲学
每个 Agent 都是一个在特定领域不断进化的独立大脑，但它们都效忠于同一个“人类档案”。
